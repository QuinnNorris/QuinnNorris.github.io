前端组件化解决方案React.js

11月 08, 2018

技术背景

JavaScript刚刚出现的时候，它的功能只是htmlDOM修改以及简单的动态响应。但是随着技术的发展，现在前端脚本所承担的压力越来越大，在一个项目中JS的代码也越写越多。这样就出现了一个问题，因为JS可以理解成是对html的修饰，所以这两种语言的耦合度是很高的，如果两段html有略微不同那么操作他们的JS代码也会不同，虽然JS的不同可能也是微乎其微的，但是这还是要让你ctrl+cv一遍，写出非常雷同的两段代码。简单的说，耦合性造成了我们写下了很多重复度很高的代码。如何化繁为简？一种思路是将前端的页面组件化，一块样式对应一块固定的代码，我们称之为一块组件，将这段代码中具有耦合性的部分抽取出来单独封装。让组件可以复用，从而为编写者减负，这就是React.js的核心思想。

如何利用React.js搭建前端

React.js提供了一种组件的概念，只要我们合理的利用组件，就可以很大程度上减轻工作量。React作为一种基本的库，有很多框架是以此库为基础并提供了常用的控件的代码。这里使用的是AntD作为框架，这个框架是基于React的。另外一提的是，这里也有使用TypeScript，这是一种在团队开发上比JS更和善的类JS语言，主要是为JS中不受限制的类型做规定，减少很多不必要的歧义。TypeScript也是基于ES6的。

一个简单的React实例

  • 通过JSX语言我们直接在render中用html的格式写出这块组件所体现的效果
  • render中必须要用一个外层元素把内容进行包裹
  • 类名可以作为标签名出现，首字母需要大写
  • state的主要作用是用于组件保存、控制、修改自己的可变状态
  • state中状态可以通过setState方法进行更新，setState会导致组件的重新渲染
  • props的主要作用是让使用该组件的父组件可以传入参数来配置该组件
  • 除非外部组件主动传入新的props，否则组件的props永远保持不变
  • 尽量少地用state，尽量多地用props

组件渲染周期

因为React的逻辑是将组建的html样式放在render中，然后通过挂载的方式渲染到html页面中，那么这其中必有一个渲染的过程。如果有渲染的过程，也就必然有渲染前、后、取消渲染的过程。这些过程都被React捕捉到了，并且实现了方法，我们可以通过重写这些方法来在我们想要的地方进行操作。

  • componentWillMount：组件挂载开始之前，也就是在组件调用 render 方法之前调用
  • componentDidMount：组件挂载完成以后，也就是 DOM 元素已经插入页面后调用
  • componentWillUnmount：组件对应的 DOM 元素从页面中删除之前调用

redux

使用React的时候通常会伴随使用redux，这里简单的介绍一下redux是什么。

在使用React的时候需要使用到state，这表示是一个状态，有的时候不仅当前的组件需要使用这个状态，这个组件的子组件也需要使用它。这时候我们就把这个状态作为props传入下一个组件中，但是如果我们这个状态非常通用，几乎所有的组件都需要使用，那么此时所有的组件都传一遍显然是不好的做法。我们在React中有一个context的说法，这个变量可以让所有的组件都获取到，我们可以把这个状态存入context中，这样解决了这个问题。可是随之而来的又有一个问题，如果所有的组件都能访问到context，如果他们都对其进行修改，会导致错误，所以需要一个单点修改的实现。这就是redux。

应用中所有的state都以一个对象树的形式储存在一个单一的store中。惟一改变state的办法是触发action，一个描述发生什么的对象。为了描述action如何改变state树，你需要编写reducers。

一些其他技巧

在我们详细的说明如何使用React-redux之前，我们先看一些其他的React技巧，其中不乏我们在下面仍然会使用到的。

  • 在JSX中使用大括号，大括号中可以放任何React的变量、语法，我们向其中写入逻辑，使用map遍历列表数据并用html包含数据进行展示渲染，JSX可以做到
  • 如果我们想进入页面以后自动focus到某个输入框需要使用input.focus()。React提供了ref属性来帮助我们获取已经挂载的元素的DOM节点，你可以给某个JSX元素加上ref属性
  • 可以使用dangerouslySetHTML，让你的代码可以插入原生的html语句
  • 高阶组件做一层包裹，可以包含可复用的组件，让下层的组件能够复用，而自己与其他逻辑结合，保持子组件的可复用性
  • context，最高层组件调用getChildContext方法，添加stats，让它所有的子孙组件都能访问到这个状态

实现一个基本的redux

层层递进的思考，redux模式是如何形成的：

  • 如果每个组件都有修改状态的方法，安全性差
  • 我们需要一个dispatch方法，为这个方法传入“把哪个状态改成哪个状态”的信息，让这个函数代为操作
  • 我们创建一个叫做createStore的方法，这个方法传入要所有的状态和reducer，这里的reducer做的就是“给状态和行动，根据行动改变状态”这个方法不受任何其他事物的干扰，只要参数确定了，这个方法的结果是固定的。createStore返回两个函数组成的对象，一个函数是获取状态的get函数，另一个是传入行动（行动在一般情况下只改变成为的状态），根据行动调reducer的dispatch方法。createStore方法为我们提供了“一个原状态的包装和一个执行修改的引擎，这个引擎需要我们自己传入行动”。createStore的意义在于提供了一层封装，让其他的APP也可以使用这种模式，并且之后还有一些监听的功能，这种功能如果直接放在dispatch里面比较复杂，但是放在createStore中是比较合理的。
  • 我们通过上面的方法解决了状态更改的问题，但是状态更改之后我们通常要重新渲染页面，这个时候我们需要手动调用渲染的方法，这样比较复杂，我们这里推荐使用观察者模式在createStore返回值一个subscribe方法，传入自己就可以添加为观察者
  • 这样我们又解决了自动渲染的问题，但是又有一个问题。如果我们每次都要渲染，即使新的改动并不涉及到渲染，我们还这样做就会耗费很大资源且不必要，所以我们要在渲染之前做个判断，现在的状态是不是和上一次一样。因为我们知道，一个对象如果只改变其中的字段内容，对象引用不变则==是判断不出来的（如果JAVA我们可以使用equals）。这里我们使用一个比较复杂的方法，我们在每次进行需要页面重新渲染的更新的时候，在更新的代码里采用…var的语法对主对象和需要更新的对象浅拷贝，这样不影响渲染的字段不会触发重新渲染，需要渲染的字段可以正确触发
  • 最后我们发现，state参数和reducer其实可以合二为一，因为state初始化可以在reducer中完成，这样就形成了reducer的最终形态
